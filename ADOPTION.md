# Adopting RENATA in an existing project (brownfield)

> 🇧🇷 [Versão em português](ADOPTION.pt-br.md)

> You already have a codebase — maybe months or years of it — and want the method on top of it: tech specs, ADRs, a PRD, the whole map. This guide covers that path. It replaces the old "Step 0.5 — Retrofit" of the tutorial.

---

## New project or existing project?

- **New (greenfield):** this guide is not for you. Follow [`GETTING-STARTED.md`](GETTING-STARTED.md) from Step 0 — `/renata:init`, then `/renata:prd`.
- **Existing (brownfield):** keep reading. One command orchestrates the whole adoption: `/renata:adopt`.
- **Only want the technical layer** (ADRs + code-pattern doc of a starter/repo, no product docs)? Skip adopt and run `/renata:extract-pattern <path>` directly — see [sub-step 6.7 of the tutorial](GETTING-STARTED.md#67-optional-distill-the-pattern-of-a-boilerplaterepo).

## The one command: `/renata:adopt`

`/renata:adopt` reverse-engineers your codebase into RENATA artifacts, **stage by stage, confirming every item with you**. The rule throughout: *the code is the truth, the human validates item by item.* It never writes code, never "fixes" your pattern, and never marks anything as verified — that's your job, via `/renata:status`.

| Stage | What it generates | What you confirm |
|---|---|---|
| 0 · Pre-flight | Scaffold via `/renata:init` if missing; detects code scopes | The list of scopes (`frontend/`, `backend/`, `src/`…) |
| 1 · Technical pattern | ADRs + `code-pattern-<scope>.md` + `rules.yaml` (composes with `/renata:extract-pattern` per scope) | Every pattern item, one by one ("possible hack — does it become a rule?") |
| 2 · Technical context | `stack.md` + `arquitetura.md` (C4 levels 1-2) | The C4 diagrams and inferred boundaries |
| 3 · Feature inventory | `docs/features/README.md` — 3-9 user-facing features found in routes/screens/modules | Each feature: keep / rename / merge / discard |
| 4 · Retroactive PRD | `docs/prd/<slug>.md` — the bet the code embodies, inferred and confirmed section by section | Problem, persona draft, hypothesis, scope, candidate metric |
| 5 · As-built specs | `docs/features/F<N>-<slug>.md` for the **selected** features (anchor ⚓ + what you'll touch next) | The selection, then each spec |
| 6 · Wrap-up | `CLAUDE.md` updated + the artifact map below + pointers to what it did NOT create | — |

Every generated artifact carries an **as-built provenance mark** right below the title:

```markdown
> 🏗️ As-built — reverse-engineered from code by /renata:adopt on <date>. Verify with /renata:status.
```

That mark means: real content, born from code, **not yet verified by a human**. In `/renata:status` terms, the step is 🔄 — you verify it and stamp it ✅ yourself.

## Where every artifact lives

The map that answers "where do the tech specs go?":

| Artifact | Where it lives | Created by | Verified via `/renata:status` |
|---|---|---|---|
| Retroactive PRD | `docs/prd/<slug>.md` | adopt Stage 4 (or `/renata:prd`) | Step 2 |
| Feature inventory (index, anchor ⚓) | `docs/features/README.md` | adopt Stage 3 (or `/renata:feature-breakdown`) | Step 7 |
| Feature specs (as-built) | `docs/features/F<N>-<slug>.md` | adopt Stage 5 (or `/renata:feature-spec`) | Step 8 |
| Code pattern — "how the code does it" | `docs/technical-context/code-pattern-<scope>.md` | `/renata:extract-pattern` (adopt Stage 1) | — |
| Stack | `docs/technical-context/stack.md` | adopt Stage 2 | Step 10 |
| Architecture (C4) | `docs/technical-context/arquitetura.md` | adopt Stage 2 | Step 10 |
| ADRs — "why we decided" | `docs/decisions/ADR-NNN-<slug>.md` | `/renata:extract-pattern` / `/renata:adr` | Step 6 |
| ADR enforcement | `.claude/rules.yaml` | `/renata:extract-pattern` / `/renata:adr` | — |
| Personas / journey / metrics | `docs/business-context/*.md` | **NOT generated by adopt** → `/renata:persona`, `/renata:user-journey`, `/renata:metrics` | Steps 3-5 |
| Progress map | `.claude/progress-map.yaml` | `/renata:init` | — |

> **"Tech spec" in RENATA is two different things.** The spec of a *feature* (what it does, done criteria, phased plan) lives in `docs/features/F<N>-<slug>.md`. The spec of the *pattern* (how the code does things: structure, stack, conventions) lives in `docs/technical-context/code-pattern-<scope>.md`, governed by ADRs in `docs/decisions/`. `/renata:adopt` generates both.

## Before you run it

1. **Install the plugin** (once): `/plugin marketplace add AInsteinsBR/renata` + `/plugin install renata@ainsteins`.
2. **Have git** in the project — run `git init` first if needed. The ADR-enforcement hook only activates with git (adopt warns you, but doing it first avoids a second pass).
3. That's it. You do NOT need to run `/renata:init` first — adopt's pre-flight runs it if the scaffold is missing. (If you did run `/renata:init` on a directory with code, it detects the brownfield and points you here.)

Then, in the project directory:

```text
/renata:adopt
```

or, naming the scopes yourself:

```text
/renata:adopt frontend/ backend/
```

## What a run looks like (condensed)

```text
/renata:adopt

Stage 0 · Found scaffold: missing → running /renata:init flow… done.
         Detected scopes: src/ (Node/Express). Confirm? → you: yes

Stage 1 · @pattern-mapper scanned src/. 14 items across 4 axes.
         "Express + Prisma (strong, seen in 12 files)" → you: confirm
         "Raw SQL in reports.ts (weak — possible hack?)" → you: not a rule
         → ADR-001..004 written, code-pattern-src.md written, rules.yaml updated.

Stage 2 · stack.md + arquitetura.md (C4 L1-L2) drafted → you review the diagrams: ok.

Stage 3 · Found 3 features: auth (routes/auth.ts), task CRUD (routes/tasks.ts),
         CSV export (routes/export.ts). Keep/rename/merge? → you: keep all
         → docs/features/README.md written (all MUST, hypothesis TBD).

Stage 4 · Draft PRD from README + inventory. Problem (guess — confirm): "freelancers
         lose billable hours to task tracking"… → you correct section by section
         → docs/prd/taskflow.md written; anchor ⚓ marked on task CRUD.

Stage 5 · Spec now for: ⚓ task CRUD + export (you'll touch it next)? → you: yes
         → F1-task-crud.md, F3-csv-export.md written (as-built).

Stage 6 · ✓ RENATA adopted. Every artifact is 🔄 — run /renata:status.
```

Every artifact above was born with the as-built mark and **without** the `> ✅ Verified by you` line — that line only you add, through the status gate.

## After adopt: reading `/renata:status`

Right after a full run, the map looks like this:

- **🔄 Steps 1, 2, 6, 7, 8, 10** — scaffold, PRD, ADRs, features, specs, architecture: artifacts exist (as-built) and await your verification. Verify them **in prereq order** (status points the way): 1 → 2 → 6 → 7 → 8 → 10.
- **⬜ Steps 3, 4, 5** — personas, journey, metrics: adopt does NOT invent these; the code doesn't reveal who the persona is or which number matters. Run `/renata:persona`, `/renata:user-journey`, `/renata:metrics` — this is where the retroactive PRD's `TODO(measure)` baselines get real.
- **⬜ Steps 7.5, 9, 11+** — phasing, roadmap, plans: in brownfield you plan the **next** phase, not the past. When you have new work, continue the normal flow (`/renata:phase-roadmap` → `/renata:feature-spec` → `/renata:plan-phase` → `/renata:execute`).

## Projects with pre-existing ADRs

If you already had ADRs before adopting:

- adopt (and extract-pattern inside it) switches to **refine mode** — it never overwrites them.
- If `.claude/rules.yaml` was populated **outside** the `/renata:adr` flow (manually, or left empty), that's technical debt: the YAML lost its binding to the ADR. For each existing ADR that has hook enforcement, run:

```text
/renata:adr refinar ADR-001 — re-validar enforcement no rules.yaml
```

`/renata:adr` in refine mode reads the ADR, validates the corresponding YAML block, and completes what's missing with your confirmation.

## FAQ

**Where do the tech specs live?**
Feature specs: `docs/features/F<N>-<slug>.md`. Pattern/stack/architecture: `docs/technical-context/`. Decisions: `docs/decisions/`. Full map in ["Where every artifact lives"](#where-every-artifact-lives) above.

**Does adopt overwrite my README, code or existing docs?**
No. It never touches code, and pre-existing RENATA artifacts (PRD, ADRs, specs) switch it to refine mode. `/renata:init`'s scaffold copy asks before overwriting `CLAUDE.md`.

**Can I re-run it?**
Yes. Re-running detects what already exists and refines instead of recreating — useful to adopt one scope now (`/renata:adopt frontend/`) and another later.

**Do I have to spec every feature?**
No — that's deliberate. The inventory covers 100% of the features; full specs only for the anchor + what you're about to touch. The rest: `/renata:feature-spec F<N>` on demand, right before working on it.

**My legacy project already has its own established method. Should I adopt?**
Probably not — see Appendix A of [`REFERENCE.md`](REFERENCE.md). Adoption is for codebases *without* a working method.
